# GitHub Rust 社区情感分析项目需求说明

## 1. 项目目标

构建一套面向 GitHub Rust 社区的自动化数据流水线，每周采集指定仓库中的 Issue、Pull Request（PR）及评论，经过清洗后使用 DeepSeek 生成方面级情感标签，并在后续接入训练好的 BERT 模型，对新增语料自动标注。

项目最终部署在 Ubuntu 两核 4 GB 服务器上；BERT 训练在 V100 GPU 环境中完成，生产服务器只负责数据采集、定时任务、数据存储和模型推理。

## 2. 当前范围

当前优先完成后端数据流水线，包括：

1. GitHub 数据采集；
2. 文本清洗与统一语料生成；
3. DeepSeek 标注；
4. BERT 标注接口与模块预留；
5. MySQL 数据存储；
6. 每周自动执行与运行记录。

当前暂不实现前端和对外 API。未来前端仅作为只读数据展示页面，用户不能主动触发采集、清洗或标注任务。

## 3. 每周数据流程

### BERT 训练完成前

```text
GitHub 增量采集
→ 原始数据写入 MySQL
→ 文本清洗与去重
→ DeepSeek 标注
→ 人工抽样检查
→ 导出训练数据
→ 在 V100 上训练 BERT
```

### BERT 训练完成后

```text
GitHub 增量采集
→ 文本清洗与去重
→ BERT 自动标注
→ 结果写入 MySQL
```

DeepSeek 后续主要用于标注训练样本、抽查 BERT 结果，以及处理 BERT 低置信度语料，不必持续标注全部新增数据。

## 4. GitHub 采集需求

采集对象为配置的 Rust 仓库白名单，不在代码中固定仓库数量或名称。

原始数据分为四类：

- Issue；
- Pull Request；
- Issue comment；
- PR comment，其中通过 `comment_type` 区分普通讨论评论与 Review comment。

至少保存以下信息：

- GitHub ID、仓库、来源类型和父级 ID；
- 标题、正文、作者、状态；
- 创建时间、更新时间、关闭或合并时间；
- GitHub URL；
- 采集时间。

爬虫必须支持：

- 首次历史数据回填；
- 每周增量更新；
- 分页、限流等待、超时和重试；
- 幂等写入，重复运行不产生重复记录；
- 中断后继续处理；
- 记录每次运行的状态、数量和错误。

### 4.1 历史回填与增量更新

- **历史回填**：项目首次运行时，从 GitHub 分页获取指定仓库已有的历史数据并写入 MySQL；
- **增量更新**：历史回填完成后，每周只获取上次成功任务之后新增或更新的数据；
- 历史回填中断后应能继续执行，已写入的数据通过唯一键和 upsert 避免重复；
- 不使用固定 `max_pages` 截断数据，而是根据 GitHub 返回的分页链接持续获取，直到没有下一页。

### 4.2 请求与分页策略

- 使用带连接复用的 HTTP Client，并通过 GitHub Token 发起认证请求；
- 每页请求 `per_page=100`，按照 GitHub `Link` 响应头获取下一页；
- 默认串行请求，优先保证稳定性并避免 GitHub 次级限流；
- 首次历史回填如确有性能需要，可配置最多两个仓库并行；
- Issue 与 PR 通过仓库 Issue 列表统一增量获取，再根据响应中的 `pull_request` 字段分类；
- Issue 和 PR 的普通评论使用仓库级 Issue comments 接口获取，避免对每个 Issue/PR 单独请求；
- Review comment 使用仓库级 PR review comments 接口获取；
- PR 详情只对本次新增或更新且确有详情字段需求的 PR 请求，避免不必要的逐条调用。

### 4.3 增量游标

每个仓库至少分别保存以下增量游标：

- Issue/PR 的最后更新时间；
- Issue/PR 普通评论的最后更新时间；
- Review comment 的最后更新时间。

增量请求按照 `updated_at` 升序处理。为避免时间精度、接口延迟或同一秒多条记录造成遗漏，实际请求的 `since` 时间应比已保存游标向前回退一个小的安全窗口，默认 5 分钟。回退产生的重复数据由数据库 upsert 消除。

游标只能在对应数据流的所有分页均成功处理后推进。如果中间页面失败，已成功页面可以保留在数据库中，但不得更新游标；下次运行重新获取该时间段并依靠幂等写入去重。

### 4.4 限流、错误与重试

- 每次请求检查 `X-RateLimit-Remaining`、`X-RateLimit-Reset` 和 `Retry-After`；
- 网络超时和可恢复的 5xx 错误使用带随机抖动的指数退避重试；
- 遇到 429、主限流或次级限流时，严格按照 GitHub 响应头暂停请求；
- 401、无权限的 404、参数错误等不可恢复错误应记录并结束对应任务，不进行无限重试；
- API 请求失败不得被当成“没有更多数据”，也不得静默跳过；
- 每个仓库和数据流均记录请求页数、读取数量、写入数量、重试次数和最终状态。

### 4.5 写库与任务恢复

- GitHub 每页数据解析后，使用 MySQL 批量 upsert 写入，一页对应一个小事务；
- 原始表以 GitHub ID（必要时加来源类型）建立唯一约束；
- 每周任务启动前检查是否已有同类型任务运行，避免重复调度；
- 单页或单仓库失败不能破坏其他已提交数据；
- 下次运行通过任务状态、增量游标和未完成记录继续处理。

### 4.6 分批与分片

GitHub API 分页和 MySQL 批量写入仅作为处理批次，不生成分页 JSON 文件，也不需要后续合并：

```text
GitHub 一页数据
→ 解析
→ 批量 upsert 到 MySQL
→ 请求下一页
```

LLM 标注同样从 `corpus` 表分批读取未标注语料并直接写回数据库，不生成待合并的标注分片。只有未来导出大规模 BERT 训练集时才按需生成多个数据文件，训练程序可直接读取这些文件。

## 5. 数据存储

数据库使用 MySQL。第一版采用以下核心表：

- `repositories`：采集仓库配置；
- `issues`：Issue 原始数据；
- `pull_requests`：PR 原始数据；
- `issue_comments`：Issue 评论；
- `pr_comments`：PR 普通评论与 Review comment；
- `corpus`：统一的清洗后语料；
- `llm_annotations`：DeepSeek 标注结果；
- `bert_predictions`：BERT 预测结果；
- `pipeline_runs`：每周流水线运行记录。

原始数据按 GitHub 对象分表保存，清洗和标注阶段统一通过 `corpus` 表处理。LLM 标注与 BERT 预测分别存储，不互相覆盖。

## 6. 统一语料与上下文

`corpus` 至少保存：

- `source_type`、`source_id`、`parent_id`；
- `raw_text`：不可变原文；
- `context_text`：辅助理解的上下文；
- `target_text`：实际需要标注的文本；
- `model_input`：按固定模板拼接后的模型输入；
- `clean_text`、语言、内容哈希和清洗规则版本。

不同来源的模型输入规则如下：

- Issue：标题与正文均作为标注目标；
- PR：标题与正文均作为标注目标；
- 普通评论：父级 Issue/PR 标题作为上下文，当前评论作为目标；
- Review comment：PR 标题作为上下文，当前评论作为目标；文件路径仅在原始表中保留，不传入模型。

第一版不传入完整讨论串或完整代码 diff，避免上下文过长及其他用户的情感污染当前评论。LLM Prompt 必须明确：上下文只辅助理解，只标注目标文本表达的情感。

## 7. LLM 标注规范

DeepSeek 输出 `aspect → class`，标准结构为：

```json
{
  "annotations": [
    {
      "aspect": "performance",
      "class": "positive"
    }
  ]
}
```

允许的 class：

- `positive`；
- `neutral`；
- `negative`。

未涉及的 aspect 不出现在 `annotations` 中。`neutral` 表示文本明确讨论该方面，但没有明显正面或负面态度。

第一版 aspect 共 16 个：

- Language：`ownership`、`type_system`、`safety`、`performance`；
- Experience：`learning_curve`、`compile_time`、`error_message`、`debugging`；
- Engineering：`maintainability`、`readability`、`extensibility`、`api_design`；
- Ecosystem：`package_manager`、`libraries`、`framework_support`、`community`。

标注结果必须经过结构校验，并保存标签体系版本、Prompt 版本、DeepSeek 模型名称、原始响应、解析结果和错误信息。标签体系变化后允许基于保留的原始语料重新标注。

## 8. 项目分包

```text
github-sentiment/
├── crawler/          # GitHub 数据采集
├── llm_labeler/      # DeepSeek 标注
├── bert_labeler/     # BERT/ONNX 推理
├── storage/          # MySQL 表模型与数据读写
├── pipeline.py       # 串联每周流水线
├── config.py         # 配置与环境变量
└── tests/            # 自动化测试
```

各模块只通过明确的数据结构交互，数据库读写统一放在 `storage` 中，避免爬虫和标注模块分别维护数据库逻辑。

## 9. 调度与部署

- 使用 Ubuntu `cron` 或 `systemd timer` 每周自动运行一次；
- 不引入 Celery、Redis、RabbitMQ 或 Kafka；
- GitHub Token、DeepSeek API Key 和数据库密码仅通过环境变量配置；
- 每一步均需幂等，单步失败不能破坏已完成数据；
- 失败任务和未处理语料可在下一次运行时继续；
- 生产服务器不承担 BERT 训练，只加载训练完成的模型执行推理。

## 10. 第一阶段验收标准

1. 能从仓库白名单完成历史回填和每周增量采集；
2. 重复执行不会产生重复数据；
3. 四类 GitHub 数据能正确关联到统一语料；
4. 评论标注时能正确携带标题上下文并区分上下文与目标文本；
5. DeepSeek 输出通过白名单和 JSON 结构校验后写入 MySQL；
6. 任意标注结果均可追溯到原始文本、清洗版本、标签体系、Prompt 和模型；
7. 流水线中断后能够继续运行，并可查询最近一次执行结果。
